Phase 0 Do This First
Fix the Broken Foundation
You can't improve a system that's failing. Fix the 401 errors and duplicate heartbeats first.

What this means in plain English

Morning Briefing and Memory Audit cron jobs have been failing with "401 Unauthorized" errors. The heartbeat was firing 2–3x per interval. Running an improvement loop on top of these failures means the loop would be optimizing a broken baseline — like tuning an engine that has a coolant leak. Fix the plumbing first.

    Fix Anthropic 401 auth errors on Morning Briefing and Memory Audit. Diagnose root cause (token expiry vs. config drift). Verify with 3 consecutive successful runs.
  • 2
    Fix duplicate heartbeat — add a 30-second dedup guard to heartbeat-state.json so two firings within 30s are collapsed to one.
  • 3
    Clean 48-hour cron run — all jobs succeed, no duplicates, no auth errors. This is the Phase 0 exit gate.

Phase 1 After Phase 0
Instrumentation — Mine Corrections, Build Eval Harness
Turn 30 days of your corrections into a test battery. This is the val_bpb equivalent.

What this means in plain English

A sub-agent reads every daily memory file from the last 30 days and extracts every correction you've made. "Use PT time," "stop running on weekends," "be more concise" — each becomes a labeled test case. The eval harness is just those test cases in a format that can be run automatically. Once a skill has 5+ test cases, it can enter the improvement loop.

  • A
    Mine corrections from memory files — sub-agent reads 30 days of daily logs, extracts ~15–25 labeled corrections. Saves to memory/correction-corpus.json with categories: format, tone, missed context, wrong priority, hallucination, config.
  • B
    Build eval runner (runs parallel with A) — a small JS script that takes a skill file + test cases, runs each through a Sonnet call, scores pass/fail. Returns aggregate score. Estimated: ~30 lines of code, costs $0.02/run.
  • C
    Wire correction capture going forward — ClawdMode detects when you're correcting behavior (keywords: "don't," "stop," "wrong," "next time") and logs to correction-inbox.json automatically. Weekly cron converts new corrections into test cases.
🚦
Gate to Phase 2: At least 1 skill has ≥5 test cases AND the eval runner produces consistent scores on repeated runs (no flaky tests).
Phase 2 Runs Automatically
Skill Improvement Loop — The Autoresearch Engine
Every night: propose a skill edit → test it → promote if better → report in morning briefing.

What this means in plain English

After you go to sleep, a sub-agent reads each skill file that has ≥5 test cases. It proposes a specific improvement, writes the edit to a staging file (never the live file), runs the eval harness against the staged version, and compares scores. If the new version passes all existing tests and scores higher, it's promoted automatically. You wake up to a changelog in your morning briefing. If you don't like a change, reply "revert heartbeat" or whatever skill — and it's undone.

  • 1
    Nightly sub-agent run — max 10 sequential sessions, one per skill. Each session: read skill → propose edit → write to SKILL.staging.md → run eval → compare → promote or discard.
  • 2
    Auto-promotion rule — staging version promotes automatically IF: (a) aggregate eval score strictly improves, AND (b) zero individual test case regressions. Otherwise discarded. Circuit breaker: 3 consecutive discards → skill paused, Mike alerted.
  • 3
    Morning briefing changelog — every morning's briefing includes: skills improved, skills paused, new corrections captured, eval score trends. One section, 10 seconds to scan. Reply "revert [skill]" to undo anything.
📊
Validation gate: Track the first 10 auto-promotions. If you revert ≥3 of them, we switch to requiring your approval before promotion. If you revert 0–1, auto-promotion is validated and stays.
Phase 3 Always Human-Gated
Behavioral Meta-Loop — Improving SOUL.md Itself
The loop surfaces proposals for changing ClawdMode's core personality rules. You approve or reject each one.

What this means in plain English

After Phase 2 is running and proven, the loop can start looking at patterns across all corrections and propose changes to the meta-instructions — SOUL.md, HEARTBEAT.md, morning briefing prompt. These are higher-stakes changes because they affect ALL behavior, not just one skill. They are ALWAYS presented to you as proposals. ClawdMode cannot change SOUL.md without your explicit "approve" reply. Ever.

  • 1
    Prompt improvements (morning briefing, HEARTBEAT.md) — staged, presented in morning briefing as proposals. You say "approve" or ignore. Never auto-promoted.
  • 2
    SOUL.md proposals — if 8+ corrections point to the same behavioral pattern (e.g., "you're too verbose"), ClawdMode surfaces a specific proposed rule change: "Suggest adding: responses under 150 words unless Mike explicitly requests detail." You approve or reject explicitly.
  • 3
    Quarterly review — you and ClawdMode review the full correction corpus, improvement history, and skill trends. Decide whether to expand scope, tighten gates, or adjust circuit breaker thresholds.

Can we start the improvement loop before building a test harness?
Codex wins
Opus said
Start now using passive signals — if you don't correct it, it was probably fine. Don't block on building an eval harness first.
Codex said
Passive silence is dangerous for an EA agent. "Mike didn't reply" could mean nothing was wrong, OR that ClawdMode missed something Mike didn't catch either. Optimizing for silence teaches the system to be less incisive.
Verdict: Codex wins — with Opus's pragmatism on harness size. The loop must have a test harness, but it only needs 5 labeled test cases per skill (drawn from correction mining), not 50. Five cases is enough to catch directional regressions without blocking for months.
Mine corrections first, then build harness — or do both in parallel?
Both conceded
Opus said
Mine corrections first. Understand the taxonomy before building infrastructure for it.
Codex said
Run in parallel. Define the harness JSON schema first as the shared contract, then mine corrections directly into that format.
Verdict: Parallel — define the schema first (10-minute task), then mine corrections and build the eval runner simultaneously. The schema is the shared contract. Zero delay, zero quality tradeoff. Both councils conceded this.
Do skill improvements auto-promote, or does Mike review each one?
Tiered verdict
Opus said
Auto-promote if eval score improves. Mike gets a changelog and can revert, but doesn't need to approve each change.
Codex said
Human review before promotion — implied but not fully specified.
Verdict (tiered by file type):
• Skill files (SKILL.md): Auto-promote if score strictly improves. Mike gets changelog + revert option.
• Cron prompts (morning briefing, HEARTBEAT): Human-gated. Higher blast radius — bad prompt wastes every day.
• SOUL.md / AGENTS.md: Always human-approved. Never autonomous.

⚠️ Goodhart's Law — same risk as the TMS platform

If the only metric is "eval score goes up," the system can optimize for passing tests while real-world quality stays flat or drops. The early warning signal: your explicit correction rate stays flat or increases despite improving eval scores, OR you start reverting auto-promoted changes. If that happens: pause all loops, rebuild test cases from fresh corrections, and treat the system as a suggestion engine (not autonomous) until calibrated.

Questions
Things you might be wondering
How is this different from you just updating MEMORY.md when I correct you?
Right now, corrections go into MEMORY.md manually and informally. They often get lost or stale. This system: (1) captures every correction in a structured format, (2) converts corrections into repeatable tests, (3) uses those tests to propose and validate specific edits to skill files, and (4) measures whether the edit actually prevents the correction from happening again. MEMORY.md notes don't get tested. This does.
What happens if the improvement loop makes ClawdMode worse?
Three safeguards: (1) Staging paths — changes never go to live files without passing the eval harness first. (2) Circuit breaker — 3 consecutive regressions and the skill's loop pauses automatically. (3) Revert — reply "revert [skill]" in the morning briefing and the promotion is undone. The worst case is one morning of slightly worse output for one skill, then an automatic rollback.
Can this change who ClawdMode is? Tone, personality, how it talks to me?
Not autonomously. SOUL.md and AGENTS.md are hard-locked — the loop can never touch them without your explicit approval. The loop only works on individual skill files (heartbeat, weather, github, etc.) and cron prompts (with your approval). Core personality, communication style, and behavioral rules are yours to control.
Does the same system work for Clawdia (Dane's AI)?
Not yet — and deliberately so. The council ruled that multi-agent portability is post-proof. We build and validate the loop for ClawdMode first. Once it's demonstrably working (correction rate down, stable eval scores, low revert rate), the pattern generalizes to Clawdia. Trying to build for both agents from day one adds complexity before we know if the approach works.
Why does Phase 0 matter? Can't we just start improving skills?
If Morning Briefing is failing with 401 errors, running an improvement loop on the morning briefing skill produces confounded results — we can't tell if a proposed change improved things or if the cron job just happened to succeed that night. Garbage baseline = garbage eval = garbage improvements. Phase 0 is about getting clean signal before we try to learn from it.
