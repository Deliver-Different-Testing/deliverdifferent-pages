Phase 1  ·  Build this first Build Now
Signal Capture — Record Every Human Decision
Every time a dispatcher changes the AI's suggestion, we capture why. That's the training data.

What it is in plain English

Right now, when a dispatcher overrides the AI's job assignment, that knowledge disappears. We're building a simple recording system — every AI decision + what the human actually did + why — stored in a secure database. Think of it as a flight data recorder for every decision in your TMS.

What gets built

  • 1
    Decision log database — a new service that records AI suggestion + human choice + outcome for every dispatch decision. Tenant-isolated. Append-only (it never gets edited — this is the ground truth).
  • 2
    3-tap override UI in Auto-Dispatch — when you change the AI's suggestion, you tap one of three reasons: "AI was wrong" / "Business exception" / "My preference." One tap. That context is gold for training.
  • 3
    Connect the existing tools — email parser, auto-dispatch, and all of Clawdia's MCP tool calls automatically emit a signal whenever a human reviews or changes an AI outcome. No manual logging.
🚦
Gate to Phase 2: 100 real override signals captured across at least 3 customers. With less data than this, the learning loop is just guessing.
Phase 2  ·  Unlocks after Phase 1 gate After 100 Signals
Learning Loop — Turn Data Into Better Decisions
After every 100 overrides, the system analyzes what it got wrong and updates its own decision rules.

What it is in plain English

Once we have enough data from Phase 1, the system runs a "learning cycle" — it looks at every dispatch decision that a human overrode, figures out the pattern ("when it's a medical run on a Friday after 4pm, don't assign Driver X"), and writes that as a rule. The rule is tested on half your dispatches for a week. If override rates drop, the rule is promoted and becomes the new default. If they don't improve, it's discarded automatically.

What gets built

  • 1
    Learning cycle scheduler — triggers automatically every 100 new override signals per customer. Groups overrides by type, identifies patterns, generates structured decision rules (not vague notes — actual testable logic).
  • 2
    A/B rule testing — new rule applies to 50% of decisions for 1 week. If override rate drops by more than 3%, the rule wins and goes to 100%. If not, it's discarded. You never see bad rules in production.
  • 3
    Kill switch — if override rate spikes up by 5% at any time, learning cycles freeze and the last good ruleset is restored. Automatically. No human needed.
🚦
Gate to Phase 3: 2 learning cycles must complete and show measurable improvement (override rate down, job completion rate stable or up). If the learning loop isn't working, we fix it before adding the build pipeline.

Autonomous Build Pipeline — Clawdia Fixes Bugs Herself
Clawdia spots patterns that need code changes, writes the fix, tests it, and hands you a PR to review.

What it is in plain English

By Phase 3, the decision log is rich enough that Clawdia can spot "there's a code bug causing this override pattern" — not just a rule update. She writes the code fix, runs the full Playwright test suite (which already exists), and if tests pass, she opens a pull request with a risk label attached. You review and merge. You never write a ticket for routine patterns again.

What gets built

  • 1
    Weekly pattern scan — Clawdia reviews the decision log for override clusters that point to code issues vs. rule issues. Code issues go to the build pipeline. Rule issues go to the learning loop.
  • 2
    Auto-PR with risk classification — every PR Clawdia opens is labelled LOW / MEDIUM / HIGH. LOW = config change (auto-mergeable after tests). MEDIUM/HIGH = requires your review. Clear, no ambiguity.
  • 3
    Shadow deploy for risky changes — MEDIUM and HIGH risk changes run at 5% of real traffic for 3 days before full rollout. If something breaks, it breaks for 5% — not everyone.

The Risk
The one thing that could break this
The council flagged one specific failure mode worth knowing about.
⚠️ Goodhart's Law

If the system's only goal is "reduce the number of times dispatchers override the AI," it will learn to make confident-sounding recommendations that dispatchers don't bother to override — not better dispatch decisions. The AI gets lazily trusted, not genuinely better.

The Fix (built in from day one)
  • Track two metrics separately: override rate AND job outcome rate (completion, disputes, driver satisfaction)
  • If override rate goes down but job disputes go up — the system is gaming the metric. Automatic alert.
  • The event log schema includes an "outcome" field for every decision. This is how we separate "accepted but wrong" from "overridden but right."

Common Questions
Things you might be wondering
Can this break existing customers?

No. Phase 1 is purely additive — we're adding recording, not changing any existing logic. The 3-tap override UI is optional context, not a blocker. The AI still works exactly as it does today. Learning rules don't change dispatch behavior until they pass A/B testing. Every change to dispatch logic is tested before it touches real traffic.

Who owns production deployments?

You do. Clawdia can build, test, and open pull requests. She cannot merge to production without your explicit approval. This is a hard guardrail in Phase 3 — it doesn't change regardless of how confident the AI is. Compliance and billing code paths always require human review regardless of risk classification.

What does Clawdia do vs. what does ClawdMode do?

Clawdia (Dane's AI) is the product builder — she owns the three phases, builds the code, runs learning cycles, and opens PRs. ClawdMode (your AI) is the ops validator — she checks that learning cycle outputs are consistent with your customer SLAs, routes exceptions correctly, and escalates anything operational to you. Think of Clawdia as the engineering lead and ClawdMode as the customer success QA.

When does cross-tenant learning happen? (e.g., what Medical Couriers teaches applies to OTG Cargo)

Not in Phases 1–3. Cross-tenant aggregation is deferred until after Phase 3 is live, all tenants are on per-tenant learning, and a legal review of data sharing across customers is complete. There's real value in cross-tenant learning (a pattern from one medical courier applies to another), but it requires careful legal groundwork first. It's in the roadmap as a post-Phase-3 conditional item.

What if a customer has very few dispatch decisions per week?

Low-volume tenants (under 30 overrides/week) can't run per-tenant learning cycles fast enough to be useful. This is an open question the council flagged as unresolved — the answer depends on actual signal volume data we'll only have after Phase 1 runs for 30 days. Once we know actual volumes, we pick one of: longer accumulation window, vertical pooling (medical couriers learn from other medical couriers), or generalized baseline model. No decision needed yet.
